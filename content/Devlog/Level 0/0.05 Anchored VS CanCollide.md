
# Concept: Anchored VS CanCollide

## Definition

- **Anchored**: When `true`, the part does not move due to physics—gravity and collisions have no effect on it.
- **CanCollide**: When `false`, objects (including players) can pass through the part as if it's not there.

They control how parts behave physically and how they interact with the game world.


## Core Purpose

These properties define:
- Whether a part stays still (`Anchored`)
- Whether a part acts like a solid object (`CanCollide`)
You use them to make floors, invisible triggers, ghost walls, floating parts, and more.

## Key Properties / Methods

| Property           | Description                           | Use Case Example              |
| ------------------ | ------------------------------------- | ----------------------------- |
| Anchored = true    | Part doesn't fall or move             | Static platforms, UI holders  |
| Anchored = false   | Part can fall or be pushed by physics | Physics puzzles, ragdolls     |
| CanCollide = true  | Part blocks player and objects        | Walls, platforms              |
| CanCollide = false | Part is pass-through                  | Triggers, ghosts, decorations |

## When to Use It

- Use **Anchored** when placing buildings, UI parts, or anything that shouldn’t fall
- Use **CanCollide = false** for invisible triggers, zones, or decorative effects
- For buttons or part-based events: `Anchored` + `CanCollide` can vary based on interaction design
## Common Mistakes / Gotchas

- Forgetting to anchor platforms which makes everything fall during playtesting
- Anchoring dynamic objects like enemies, and getting weird bugs
- Making a part `CanCollide = false` then wondering why players walk through it
- Assuming both properties are the same thing when they're not
## Code Example

```lua
-- Anchoring a part
workspace.MyPart.Anchored = true

-- Making a wall solid
workspace.MyWall.CanCollide = true

-- Invisible trigger zone
workspace.Trigger.CanCollide = false
workspace.Trigger.Transparency = 1
```
